Section 01 · Introduction

Layout in SwiftUI

Before you can build a real iOS app screen, you need to know how to arrange views on screen. In SwiftUI, everything comes down to three simple containers — and once you understand them, you can lay out any interface.

Learning Objectives
  • Understand what "layout" means in SwiftUI
  • Explain why stacks are the foundation of SwiftUI UI design
  • Identify which stack types exist and their purpose
  • Understand how container views hold and arrange child views

What Is Layout in SwiftUI?

Layout refers to how views are positioned and arranged on screen. In SwiftUI, you don't drag views onto a canvas and set pixel coordinates — instead, you describe how views should be arranged relative to each other using container views called stacks.

SwiftUI's layout system is smart. It knows the screen size, calculates how much space each view needs, and places everything automatically — but it does this based on the layout rules you define through stacks.

Why Stacks Matter

Every iOS app you've ever used was built using stacks. Profile cards, tab bars, navigation headers, settings screens, chat bubbles — all of these are combinations of HStack, VStack, and ZStack.

Understanding these three containers is the most important skill in SwiftUI layout. Once you can choose the right stack and nest them together, you can build virtually any interface.

💡 Key Mental Model: Think of stacks like boxes. HStack is a horizontal box that packs items side by side. VStack is a vertical box that stacks items top to bottom. ZStack is a transparent box where items float on top of each other. You choose the box based on the direction you want to arrange things.

Section 02 · Concept

Stack-Based Layout

A stack is a container view that holds other views and arranges them according to a specific direction. Everything in SwiftUI layout is built by choosing the right stack and nesting stacks inside each other.

Learning Objectives
  • Define what a stack is in SwiftUI
  • Explain how stacks group child views
  • Describe how views can be nested inside stacks
  • Understand how combining stacks creates complex layouts

What Is a Stack?

A stack is a container view — it holds child views inside it and tells SwiftUI how to arrange them. You write views inside the stack's curly braces, and the stack takes care of positioning them on screen.

Stacks are themselves views, which means you can put a stack inside another stack. This is called nesting, and it's how complex iOS layouts are built from simple pieces.

Swift — Anatomy of a Stack
// A stack holds child views inside curly braces VStack { // ← the container / stack Text("Hello") // ← child view 1 Text("World") // ← child view 2 Button("Tap Me") { } // ← child view 3 } // ← stack ends here

Nesting Stacks

Real app layouts require combining stacks. For example: a VStack for the overall vertical layout, containing an HStack for a row of buttons. Stacks can be nested as many levels deep as needed — though deeper nesting should be kept readable.

Nested Stacks
VStack { // outer: vertical Text("My Profile") .font(.title).bold() HStack { // inner: horizontal Image(systemName: "person") Text("Khalid K.") Spacer() Image(systemName: "chevron.right") } Text("iOS Developer") .foregroundColor(.secondary) }

Do stacks automatically size themselves? ▼
Yes. Stacks calculate their size based on their children. A VStack becomes as tall as all its children combined (plus spacing). An HStack becomes as wide as all its children combined. This automatic sizing is part of SwiftUI's layout system — you describe the structure, and SwiftUI handles the math.
How deep can you nest stacks? ▼
As deep as needed — there's no hard limit. However, very deeply nested stacks can slow down Xcode previews and make code harder to read. In practice, if you find yourself nesting more than 3-4 levels deep, it's usually a sign you should extract some views into separate structs to keep the code clean.
What happens when I put nothing inside a stack? ▼
An empty stack (like VStack { }) compiles but renders as an empty, zero-size view. Xcode might show a warning. Always have at least one child view inside a stack. In practice, start by putting placeholder Text views while you build the structure, then replace them with real content.
Section 03 · HStack

HStack
Horizontal Stack

HStack places its child views side by side, horizontally, from left to right. Use it whenever you want things arranged in a row.

What Is HStack?

HStack (Horizontal Stack) arranges its child views in a row from left to right. Think of it like a shelf — you place items on the shelf and they line up horizontally. Every tab bar, toolbar button row, and side-by-side label in iOS is built with HStack.

🧠 Quick Check — HStack
You want to build a navigation bar row with a "Back" button on the left and a "Done" button on the right, with the title "Settings" in the middle. Which combination gives you the right layout?
✅ Correct! HStack arranges items horizontally. The two Spacer() views push "Back" all the way left, "Done" all the way right, and "Settings" to the center. This is the classic navigation bar pattern used in almost every iOS app.
❌ Not quite. You need an HStack (horizontal arrangement) with Spacers between the elements to push them to their correct positions. HStack { Button("Back"); Spacer(); Text("Settings"); Spacer(); Button("Done") } is the correct pattern.
Section 04 · VStack

VStack
Vertical Stack

VStack places its child views stacked vertically, from top to bottom. It's the most commonly used stack in SwiftUI — almost every screen has at least one VStack as its backbone.

What Is VStack?

VStack (Vertical Stack) arranges its child views in a column from top to bottom. Think of it like a grocery list — each item goes below the one above it. Profile screens, article views, settings pages, sign-in forms — virtually every iOS screen uses VStack as its primary layout container.

You're building a profile card that shows a photo at the top, then the user's name below it, then their bio below that. Which stack do you use, and what alignment makes the text left-aligned?

Section 05 · ZStack

ZStack
Depth Stack

ZStack places its child views layered on top of each other, like a stack of transparent slides. The first view goes to the back, the last view goes to the front.

Learning Objectives
  • Create a ZStack and layer views front-to-back
  • Understand the back-to-front rendering order
  • Build overlays, badges, and text-on-image layouts
  • Identify when ZStack is the right choice

What Is ZStack?

ZStack (Depth Stack) arranges its child views by layering them on top of each other. The "Z" refers to the Z-axis — depth into the screen. Views declared first appear at the back; views declared later appear in front. This is how overlays, badges, full-screen backgrounds, and text-on-image effects are built.

Swift — ZStack Syntax
// Basic ZStack ZStack { Color.blue // ← renders at the BACK (bottom) Text("Hello!") // ← renders in the FRONT (top) } // With alignment — where layers align ZStack(alignment: .bottomTrailing) { Image("profile") // ← background image Circle() // ← badge at bottom-right .fill(Color.green) .frame(width: 16, height: 16) }

🧠 Quick Check — ZStack
In ZStack, if you write three views — Rectangle(), Text("Hello"), Image(systemName: "star") — in that order, which view appears on top (visible in front)?
✅ Correct! In ZStack, the order of views in code determines their depth. First declared = back (lowest layer), last declared = front (top layer). So Rectangle renders first (at the back), Text renders in the middle, and Image renders last (at the front, on top).
❌ Not quite. In ZStack, the last declared view renders at the front. So Image appears on top of Text, which appears on top of Rectangle. Think of it like painting: the last stroke covers the earlier ones.

Nesting Stacks Together

Real iOS screens combine all three stack types. A typical pattern: a VStack for the overall screen layout, containing HStacks for each row, with ZStacks for any overlaid content. Here's a real-world example:

Nested Stacks — Course Card

🧠 Quick Check — Choosing the Right Stack
You're building a "Now Playing" music card. It has: an album artwork image with a play button icon centered on top of it, a song title below the image, and the artist name below that. How do you structure the stacks?
✅ Correct! The overall layout is top-to-bottom → VStack. The artwork needs an overlay (play button on image) → ZStack. The text below stacks vertically → more VStack. Nesting ZStack inside VStack is the right approach.
❌ Not quite. Think layer by layer: the overall order is vertical (artwork, then song title, then artist) → VStack. The artwork has something on top of it (play button) → ZStack. So the answer is VStack containing a ZStack for the image+button, then the Text views.

Section 08 · Watch Out For

Common Beginner Mistakes

These are the layout mistakes that trip up almost every new SwiftUI developer. Learning to recognize them now will save you hours of debugging in your labs.

🔴 Using the wrong stack type
The most basic mistake. Wanting a vertical layout but using HStack, or wanting side-by-side content but using VStack. Before writing any code, ask yourself: "Should the views be in a row or a column?" Row = HStack. Column = VStack. Overlapping = ZStack.
🔴 Forgetting that ZStack draws items back-to-front
Students often write background colors or images after foreground content, then wonder why their text is hidden. In ZStack, first = back, last = front. Always write your background (color, image, shape) as the first item in a ZStack, and foreground content last.
🔴 Not using Spacer() to push views apart
If you want two views at opposite ends of a row (left edge and right edge), you need a Spacer() between them inside an HStack. Similarly, a Spacer() at the bottom of a VStack pushes content to the top. Without Spacer(), views cluster together in the center of the stack. Spacer() is like invisible elastic that fills all available space.
🔴 Nesting stacks the wrong way
A common confusion: putting an HStack inside an HStack when you meant to put a VStack inside an HStack. Before nesting, always ask: "What direction does the inner container go?" Then check if that matches what you wrote. Sketching the layout on paper first (even just a quick rectangle diagram) prevents this mistake.
🔴 Confusing VStack alignment with HStack alignment
In VStack, the alignment parameter controls horizontal alignment (.leading = left, .trailing = right, .center = middle). In HStack, the alignment parameter controls vertical alignment (.top, .center, .bottom). This trips students up because the parameter name is the same but it controls a different axis depending on the stack type.
🔴 Creating overly deeply nested stacks
Having a VStack inside an HStack inside a VStack inside a ZStack inside another VStack is a warning sign. Deep nesting makes code unreadable and hard to debug. When you find yourself nesting more than 3-4 levels, extract some views into separate View structs. This keeps each view focused and readable.
🔴 Using ZStack when you just need a background modifier
Not everything that appears "behind" something needs a ZStack. If you just want a background color behind a view, use the .background() modifier instead: Text("Hello").padding().background(Color.blue). ZStack is for when you need truly independent layered elements, like an image and text that don't have a direct parent-child relationship.
🔴 Missing the alignment parameter on VStack for left-aligned text
By default, VStack uses alignment: .center — all children center horizontally. This looks wrong for most content cards, articles, and forms where text should be left-aligned. Always remember to add alignment: .leading to your VStack when building content that should be left-aligned. This is one of the most frequently forgotten parameters in beginner code.
🧠 Mistake Recognition Quiz
A student writes this code and gets a layout where "Name" and "Email" appear side by side on the same line instead of one above the other. What's wrong?
HStack { Text("Name: Khalid") Text("Email: khalid@lagcc.cuny.edu") }
✅ Correct! HStack places views side by side — that's why both texts appear on the same line. To stack them vertically (Name on top, Email below), change HStack to VStack and add alignment: .leading so the text aligns to the left.
❌ Not quite. The issue is that HStack places views side by side horizontally. To get one above the other, change HStack to VStack. Adding alignment: .leading will also keep the text left-aligned, which is the expected look for profile/form content.

📚 Key Terms — Click to Flip

HStack

Click to define
Horizontal Stack. Arranges child views side by side from left to right. The alignment parameter controls vertical positioning of items within the row.

VStack

Click to define
Vertical Stack. Arranges child views top to bottom. The alignment parameter controls horizontal positioning. The most commonly used stack in SwiftUI.

ZStack

Click to define
Depth Stack. Layers views on top of each other. First declared = back layer, last declared = front layer. Used for overlays, badges, and backgrounds.

spacing

Click to define
An optional parameter on HStack and VStack that controls the gap between child views. Example: VStack(spacing: 20) { }. Default spacing is system-defined (~8pt).

alignment

Click to define
Controls how child views align perpendicular to the stack's direction. In VStack, controls horizontal alignment (.leading, .center, .trailing). In HStack, controls vertical alignment (.top, .center, .bottom).

Spacer()

Click to define
A flexible invisible view that expands to fill all available space. In HStack, creates horizontal space between items. In VStack, creates vertical space. Used to push views to edges.

✅ Perfect! The card itself needs a ZStack at the top level (gradient background layer + content layer). Inside, the post content uses VStack (avatar then name then text top-to-bottom). The avatar with its checkmark badge uses another ZStack (avatar image at back, badge circle in front, aligned to topTrailing). This is exactly how real social app cards are built.
❌ Not quite. The card as a whole needs a ZStack (gradient behind all the content). The post content flows top-to-bottom → VStack. The avatar with a badge on top needs another ZStack (avatar at back, badge in front, aligned .topTrailing). So the answer is: ZStack (card) → VStack (content) → ZStack (avatar + badge).
