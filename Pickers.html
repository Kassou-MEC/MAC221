Pickers in SwiftUI
A view that lets the user choose one value from a list of options. The cornerstone of any form-based screen.
// 04 · concept

What is a Picker?

A Picker is a SwiftUI view that lets the user choose one value from a list of options — like a dropdown, radio-button group, or spinning wheel.

⚡

Built into iOS

No extra libraries needed. One line of SwiftUI = a working picker.

🎨

3 Visual Styles

Same code can show as a menu, segmented bar, or wheel. You choose with one modifier.

🔗

Updates Automatically

The selected value is stored in a variable. SwiftUI updates it for you when the user picks.

💡 Easy way to remember: A Picker = a list of options + a variable to store what the user chose. That's it.

// 06 · easy example #1 · the simplest picker

Your first Picker — 3 ingredients

Every picker needs three things: a title, a selection binding, and a list of options. That's it. Here's the absolute simplest version.

SimplePicker.swift
● Basic
import SwiftUI

struct SimplePicker: View {
    @State private var favoriteFruit = "Apple"

    var body: some View {
        Picker("Pick a fruit", selection: $favoriteFruit) {
            Text("Apple").tag("Apple")
            Text("Banana").tag("Banana")
            Text("Cherry").tag("Cherry")
        }
    }
}
▸ What each line does
  1. @State means "this is a variable that can change. When it changes, redraw the screen."
  2. favoriteFruit = "Apple" — the starting value. Apple is selected when the screen opens.
  3. Picker("Pick a fruit", ...) — creates a Picker. The text is the label.
  4. $favoriteFruit — the $ means "connect this variable to the Picker". When the user picks, the variable changes automatically.
  5. Text("Apple").tag("Apple") — one row of the picker. The tag is what gets saved when this row is picked.
💡 Remember the $ sign: Without $, the Picker can only read. With $, the Picker can write back. We need it!

Make the label appear — wrap in a Form

A bare Picker floats in the middle of the screen and hides its label. To get the proper "label on the left, value on the right" layout you see in Settings apps, wrap the Picker in a Form.

SimplePickerInForm.swift
● Basic
struct SimplePicker: View {
    @State private var favoriteFruit = "Apple"

    var body: some View {
        Form {
            Picker("Pick a fruit", selection: $favoriteFruit) {
                Text("Apple").tag("Apple")
                Text("Banana").tag("Banana")
                Text("Cherry").tag("Cherry")
            }
        }
    }
}
▸ What each line does
  1. We added Form { ... } around the Picker. Nothing else changed!
  2. Form is a SwiftUI container. It gives us the iOS Settings-style look: gray background, white rounded rows.
  3. Now the label "Pick a fruit" appears on the LEFT and the value "Apple" appears on the RIGHT.
  4. Tapping the row opens a small dropdown menu. This is the default style inside a Form.
💡 Best practice: Real iOS apps almost always put pickers inside a Form. It looks correct and matches what users expect.

Same picker — .segmented style

We keep the exact same code from the previous slide and add one modifier: .pickerStyle(.segmented). The picker now shows as a pill bar — great for 2–4 options.

SegmentedPicker.swift
● Basic
struct SegmentedPicker: View {
    @State private var size = "Medium"

    var body: some View {
        Picker("Size", selection: $size) {
            Text("Small").tag("Small")
            Text("Medium").tag("Medium")
            Text("Large").tag("Large")
        }
        .pickerStyle(.segmented)
    }
}
▸ What each line does
  1. The Picker code is the same as before — title, binding, and three rows.
  2. The new line is at the bottom: .pickerStyle(.segmented).
  3. This tells SwiftUI: "show this picker as a segmented bar instead of a menu".
  4. The label "Size" is hidden by SwiftUI in segmented style — there's no room for it next to the bar.
⚠ When to use: Segmented works best for 2–4 short options. More than 4, or long labels, and the pills get cramped. Switch to .menu or .wheel.

// 10 · easy example #4 · cleaner with ForEach

Don't repeat yourself — use ForEach

Writing Text(...).tag(...) for 5 options is OK. For 50, it's a nightmare. Let's store options in an array and loop with ForEach.

ForEachPicker.swift
● Basic
struct FruitPicker: View {
    let fruits = ["Apple", "Banana", "Cherry",
                  "Mango", "Peach"]
    @State private var selected = "Apple"

    var body: some View {
        Form {
            Picker("Fruit", selection: $selected) {
                ForEach(fruits, id: \.self) { fruit in
                    Text(fruit)
                }
            }
        }
    }
}
▸ What each line does
  1. let fruits = [...] — an array of strings. We use let because the list never changes.
  2. Form { ... } — same as before, gives us the iOS row look.
  3. ForEach(fruits, id: \.self) — this is a loop. It goes through each fruit in the array.
  4. id: \.self tells SwiftUI: "use the fruit name itself as the unique ID". Required so SwiftUI can track which row is which.
  5. { fruit in Text(fruit) } — for each fruit in the loop, create a Text row with that fruit name.
⚠ Gotcha: If id: \.self is missing and your items aren't Identifiable, SwiftUI won't know how to track them and the picker will silently stop updating.

// 11 · easy example #5 · numbers with a range

Picking numbers — ranges made easy

Sometimes you need the user to pick a number: age, quantity, rating, etc. Instead of an array, we can loop over a range.

NumberPicker.swift
● Basic
struct AgePicker: View {
    @State private var age = 18

    var body: some View {
        Picker("Age", selection: $age) {
            ForEach(1..<100, id: \.self) { number in
                Text("\(number) years").tag(number)
            }
        }
        .pickerStyle(.wheel)
    }
}
▸ What each line does
  1. @State private var age = 18 — this time the variable is a number (Int), not a String. Default is 18.
  2. 1..<100 — a Swift range. Means "every number from 1 to 99". Use 1...100 if you want 100 included.
  3. ForEach(1..<100, id: \.self) — loop through every number in the range.
  4. Text("\(number) years") — the \(...) inserts a variable into a string. Result: "18 years".
  5. .tag(number) — saves the number when this row is picked. The tag type (Int) MUST match the variable type (Int).
💡 Important rule: The type inside .tag() must MATCH the type of the @State variable. Mixing types = the picker won't work.

// 12 · level up · type-safe with enums

Pickers + Enums

Using strings like "Easy" is dangerous — one typo and your code breaks silently. Better solution: use an enum.

💡 What is an enum? An enum (short for "enumeration") is a Swift type that represents a fixed group of related values. Instead of writing strings like "small", "medium", "large" all over your app, you define them once as cases of an enum. Then Swift makes sure you never make a typo — if you write .smll by mistake, Xcode shows a red error immediately.
EnumPicker.swift
● Intermediate
enum PizzaSize: String, CaseIterable, Identifiable {
    case small  = "Small 🍕"
    case medium = "Medium 🍕🍕"
    case large  = "Large 🍕🍕🍕"

    var id: String { rawValue }
}

struct PizzaOrder: View {
    @State private var size: PizzaSize = .medium

    var body: some View {
        Form {
            Section(header: Text("Size")) {
                Picker("Size", selection: $size) {
                    ForEach(PizzaSize.allCases) { option in
                        Text(option.rawValue).tag(option)
                    }
                }
                .pickerStyle(.segmented)
            }

            Section {
                HStack {
                    Text("Your size")
                    Spacer()
                    Text(size.rawValue)
                        .foregroundColor(.blue)
                }
            }
        }
    }
}
▸ What each line does
  1. enum PizzaSize — creates a new enum with three options: small, medium, and large.
  2. : String — each case has a String value. So PizzaSize.small.rawValue gives us "Small 🍕".
  3. : CaseIterable — gives us PizzaSize.allCases (a list of all cases). Perfect for ForEach.
  4. : Identifiable — needed by ForEach to identify each case. We use rawValue as the id.
  5. @State var size: PizzaSize = .medium — the variable type is the enum itself. No more typos!
  6. Form { Section(header: ...) } — Section adds the "SIZE" title above the segmented bar.
  7. ForEach(PizzaSize.allCases) — loops through small, medium, large automatically.
  8. The second Section shows "Your size · Medium 🍕🍕" using HStack + Spacer() to push the value to the right.
⚠ Important rule: .pickerStyle(.segmented) ALWAYS hides the picker's own label, even inside a Form. To show a label with segmented, wrap the Picker in a Section(header:) — this is the standard SwiftUI pattern for labeled segmented pickers.
CaseIterable → .allCases Identifiable → ForEach happy Section(header:) for label

// 13 · advanced · real-world data

Pickers with custom objects

In real apps, you rarely pick strings — you pick a User, a Product, a Country. Here's the production-grade pattern.

CountryPicker.swift
● Advanced
struct Country: Identifiable, Hashable {
    let id = UUID()
    let name: String
    let flag: String
    let code: String

    // Our 4 countries — defined inside Country for easy access
    static let morocco = Country(name: "Morocco", flag: "🇲🇦", code: "+212")
    static let usa     = Country(name: "USA",     flag: "🇺🇸", code: "+1")
    static let france  = Country(name: "France",  flag: "🇫🇷", code: "+33")
    static let japan   = Country(name: "Japan",   flag: "🇯🇵", code: "+81")

    static let all = [morocco, usa, france, japan]
}

struct CountryPicker: View {
    @State private var selected = Country.morocco

    var body: some View {
        Form {
            Picker("Country", selection: $selected) {
                ForEach(Country.all) { c in
                    Text("\(c.flag) \(c.name) \(c.code)")
                        .tag(c)
                }
            }
        }
    }
}
▸ What each line does
  1. struct Country — creates a custom type holding 4 pieces of info: id, name, flag, and code.
  2. : Identifiable, Hashable — these two protocols are required so we can use Country in a Picker.
  3. let id = UUID() — every Country gets a unique ID automatically. UUID = "Universally Unique ID".
  4. static let morocco = Country(...) — static means this Country belongs to the type itself. Now you can write Country.morocco anywhere in your app.
  5. static let all = [morocco, usa, france, japan] — a ready-made array of all 4 countries, accessed as Country.all.
  6. @State var selected = Country.morocco — Morocco is the default selected country.
  7. ForEach(Country.all) — loops through all 4 countries to build the picker rows.
  8. .tag(c) — saves the whole Country object when the user picks a row.
💡 Why static let is helpful: It groups your data with the type itself. Apple does this with colors: Color.red, Color.blue. Same idea here with Country.morocco.

// 14 · reference · picking the right style

Picker styles — cheat sheet

Use this table as your go-to reference. The right style depends on how many options you have and how prominent the choice should be.

StyleVisualBest ForAvoid When
.menu Tap-to-open dropdown Many options, settings rows, infrequent choice Only 2–3 options (overkill)
.segmented Horizontal pill bar 2–4 options, frequent toggle, mode switching Long labels, more than 4 options
.wheel Spinning drum Numbers, long ordered lists Quick choices — too slow to spin
.inline Expanded list, all visible Inside a Form, when all options should be seen Outside a List/Form (looks awkward)
.navigationLink Pushes a new screen Many options needing search/scroll Simple 3-option choices
.automatic SwiftUI decides for you Default — adapts to context (Form, etc.) When you need a guaranteed look
💡 Rule of thumb: Inside a Form, the default .automatic usually does the right thing. Only set a style when you want something specific (like .segmented).

DatePicker
A specialized picker built specifically for selecting dates, times, or both — with localization, ranges, and three beautiful display styles.
// 16 · concept

What is a DatePicker?

A DatePicker is a specialized SwiftUI view that helps users pick a date, a time, or both. It handles calendars, time zones, and locales — so you don't have to.

📅

Date / Time / Both

Use displayedComponents to choose: .date, .hourAndMinute, or both.

🎨

3 Visual Styles

.compact (small bubble), .graphical (full calendar), .wheel (spinning drum).

🌍

Auto-Localized

Calendars, weekdays, AM/PM, and number formats follow the user's locale automatically.

🔒

Range-Aware

Limit selectable dates with a ClosedRange<Date> — no past dates, no far future.

💡 Big idea: DatePicker is just like Picker — but its options are dates instead of arbitrary values. The selection binding is a Date.

Your first DatePicker — .compact

Just like Picker, you need three things: a title, a binding to a Date, and that's it. The compact style is the default — small and clean.

CompactDatePicker.swift
● Basic
import SwiftUI

struct BirthdayView: View {
    @State private var birthday = Date()

    var body: some View {
        DatePicker(
            "Birthday",
            selection: $birthday,
            displayedComponents: .date
        )
        .datePickerStyle(.compact)
    }
}
▸ What each line does
  1. @State var birthday = Date() — creates a variable holding today's date. Date() with no arguments = right now.
  2. DatePicker("Birthday", ...) — creates the date picker. The title is the label.
  3. selection: $birthday — connects the variable to the picker. The $ is required.
  4. displayedComponents: .date — show only the date, no time. Options: .date, .hourAndMinute, or both.
  5. .datePickerStyle(.compact) — small bubble that pops open a calendar when tapped.
💡 Space-saver: Compact is perfect inside a Form. It stays small until the user taps to open the calendar.

// 22 · advanced · limit valid dates

Restrict to a date range

Real apps need rules: "no past dates", "must be within 90 days", "between two known dates". Pass a ClosedRange<Date> as the in: parameter — SwiftUI grays out everything outside.

DateRange.swift
● Advanced
struct BookingView: View {
    @State private var checkIn = Date()

    // Valid range: today → 90 days from now
    var range: ClosedRange<Date> {
        let cal   = Calendar.current
        let start = Date()
        let end   = cal.date(byAdding: .day,
                             value: 90, to: start)!
        return start...end
    }

    var body: some View {
        DatePicker(
            "Check-in",
            selection: $checkIn,
            in: range,
            displayedComponents: .date
        )
        .datePickerStyle(.graphical)
        .tint(.pink)
    }
}
▸ What each line does
  1. var range: ClosedRange<Date> — a property that gives us back a range between two dates.
  2. let start = Date() — start of range = today (right now).
  3. cal.date(byAdding: .day, value: 90, to: start)! — adds 90 days to today to get the end date.
  4. start...end — the three dots mean "all dates from start to end". Like a number range, but with dates.
  5. in: range — tells the DatePicker: "user can only pick dates inside this range". Other dates are grayed out.
  6. .tint(.pink) — changes the accent color of the picker (the highlight color of the selected day).
⚠ Range options: start...end = both limits · start... = no past dates · ...end = no future dates.

// 27 · summary · key takeaways

What you should remember

If you only remember five things from today's lecture, make it these:

1

Picker = title + binding + options

Three ingredients. Always. The binding is what makes the picker actually do something.

2

Use enums, not strings

Strings break silently. Enums with CaseIterable + Identifiable are the SwiftUI standard.

3

Pick the right .pickerStyle

.menu for many, .segmented for 2–4, .wheel for ordered lists.

4

DatePicker is just a special Picker

Use displayedComponents for date/time and in: range to restrict.

5

Tags must match the binding type

If your binding is Country?, every tag must be Country?. The #1 silent bug.

⚡

Practice beats theory

Build the three exercises. Pickers are easy when you've written 5; impossible if you've only read about them.
