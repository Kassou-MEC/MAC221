Section 01 · Introduction

Welcome to SwiftUI

Before you build your first iOS app, you need to understand the fundamentals that everything else is built on: views, modifiers, and basic controls. This is where SwiftUI begins.

Learning Objectives
  • Understand what SwiftUI is and why it matters
  • Explain declarative UI in simple terms
  • Know why views, modifiers, and controls are the foundation of all iOS apps
  • Understand how SwiftUI connects with Xcode's live preview

What Is SwiftUI?

SwiftUI is Apple's modern framework for building user interfaces on iPhone, iPad, Mac, Apple Watch, and Apple TV. You write code that describes what your UI should look like, and SwiftUI takes care of making it happen. It was introduced in 2019 and is now the standard way to build Apple apps.

Before SwiftUI, developers used UIKit — an older, more complex system where you had to manually manage every step of how the interface was built and updated. SwiftUI replaces most of that complexity with clean, readable code that's much easier to learn.

📝

Declarative

You describe what you want the UI to look like. SwiftUI figures out how to build it and update it.

🔁

Reactive

When your data changes, the UI updates automatically. No manual refresh calls needed.

⚡

Live Preview

Xcode shows you the result of your code instantly in the Canvas — no simulator required while designing.

Declarative UI — What Does That Mean?

In traditional (imperative) code, you tell the computer every step: "Create a button. Set its color to blue. Add it to the screen. When data changes, find the button, change its label." That's a lot of manual steps.

In declarative UI, you simply describe the end result: "Show a blue button with this label." When data changes, SwiftUI automatically re-renders the interface based on your description. You never manually update UI elements — the framework does it for you.

Swift
// SwiftUI — Declarative style // You describe WHAT you want, SwiftUI handles the rest struct WelcomeView: View { var body: some View { VStack { Text("Welcome to MAC221") Text("Let's build iOS apps!") Button("Get Started") { } } } }
💡 Key Insight: Every SwiftUI view is a struct that has a body property. That body returns what appears on screen. Everything visible in an iOS app is made of views.

Why Learn This First?

Before you can build navigation, lists, forms, or animations — you need to know how to create views and style them with modifiers. And before you can lay out a screen, you need to know what basic controls look like. These two topics (3.1 and 3.2) are the absolute foundation. Everything in SwiftUI builds on what you learn in this lecture.

Section 02 · Core Concept

3.1 Views & Modifiers

A View is anything visible on screen. A modifier customizes a view. Together, they are how every single iOS interface is built in SwiftUI.

Learning Objectives
  • Define what a View is and explain the body property
  • Explain how views are composed (nested) to build interfaces
  • Apply modifiers to change a view's appearance and behavior
  • Chain multiple modifiers correctly
  • Explain why modifier order matters

What Is a View?

In SwiftUI, a View is any piece of the interface that can appear on screen. Text labels, buttons, images, text fields, dividers — they are all views. Even a container that holds other views is itself a view.

Every SwiftUI view is a Swift struct that conforms to the View protocol. That protocol requires exactly one thing: a body property that returns "some View" — in other words, it returns whatever the interface looks like.

Swift — Anatomy of a View
// Every SwiftUI screen is a struct that conforms to View struct ContentView: View { // ← conforms to the View protocol var body: some View { // ← body is required — this is what appears on screen Text("Hello, LaGuardia!") // ← the view being returned } }

Views Compose — You Build UIs by Nesting Views

SwiftUI interfaces are built by combining small views into bigger ones. You put views inside container views. A VStack (covered in 3.3) holds multiple views arranged vertically. Inside that, you can put a Text, an Image, and a Button. Each of those is its own view.

This composability means you build complex interfaces from simple, reusable pieces — just like how a house is made of bricks, not one giant slab.

Swift — Views Nested Inside Views
struct ProfileView: View { var body: some View { VStack { // ← container view (vertical stack) Image(systemName: "person.circle") // ← view 1: an icon image Text("Khalid Hassan") // ← view 2: a text label Text("iOS Developer") // ← view 3: a subtitle label Button("Follow") { } // ← view 4: a button } } }

What Are Modifiers?

A modifier is a method you call on a view to change how it looks or behaves. Modifiers are called with dot syntax immediately after a view. They return a new modified version of the view — they don't change the original view, they wrap it.

Think of modifiers like clothing layers: you start with a plain view, then add a font style, then a color, then padding, then a background. Each modifier adds another layer of customization.

Swift — Your First Modifiers
// A plain Text view — no styling yet Text("Good morning!") // Now let's add modifiers one by one: Text("Good morning!") .font(.title) // ← larger title font .foregroundColor(.orange) // ← orange text colour .bold() // ← make it bold .padding() // ← add breathing room around it .background(Color.yellow.opacity(0.2)) // ← soft yellow background .cornerRadius(10) // ← rounded corners

⚠️ Modifier Order Matters!

This is one of the most important concepts to understand. Because each modifier wraps the view and returns a new view, the order you apply modifiers directly affects the final result. SwiftUI applies modifiers from top to bottom (innermost to outermost).

Swift — Order Example A
// A: padding() THEN background() // Background fills the padded area ✅ Text("Order A") .padding(20) .background(Color.blue) // Result: blue box with space around text
Swift — Order Example B
// B: background() THEN padding() // Background only fills text area ❌ Text("Order B") .background(Color.blue) .padding(20) // Result: tiny blue box, big empty space
Result A — padding first, then background

🧠 Quick Check — Views & Modifiers
You want a Text view with orange text AND a rounded background. In what order should you apply the modifiers to get the background to include padding?
✅ Correct! Apply .foregroundColor() first (text color), then .padding() to add space, then .background() to fill the padded area, then .cornerRadius() to round it. Background applies to whatever is already there — so padding must come before it.
❌ Not quite. Apply .foregroundColor() → .padding() → .background() → .cornerRadius(). The background fills whatever size the view is at the moment it's applied — so padding must come first to expand the area.

3.2 Basic Controls

SwiftUI comes with a set of built-in controls that handle the most common interface needs. Master these four — Text, Image, Button, and TextField — and you can build most beginner app screens.

Learning Objectives

  • Display system icons and images with Image
  • Build tappable Buttons with actions and styles
  • Add user input fields with TextField
  • Combine these controls into a simple mini-UI
🔤

Text

Displays a string of text on screen. Every label, title, and body copy you see in an iOS app is a Text view.

🖼️

Image

Displays images — either from SF Symbols (Apple's built-in icon library) or from your app's Assets.

🔘

Button

A tappable element. Buttons trigger actions — submitting a form, navigating to a screen, toggling state.

📝

TextField

Lets users type input. Used for names, emails, search boxes, messages, and any text entry field.

🔤 Text

What is Text?

The Text view displays a read-only string on screen. It's the most-used view in SwiftUI — every time you need to show words to the user, you use Text. Text views support rich formatting through modifiers and can even render markdown-style attributed strings.

Swift — Text Examples

Italic text Text("Professor Khalid Hassan") .font(.subheadline) .italic() .foregroundColor(.secondary) // 7. Strikethrough (crossed out text) Text("Was $99.99") .strikethrough() .foregroundColor(.gray)

🖼️ Image

What is Image?

The Image view displays images on screen. In SwiftUI there are two main types: System Images (SF Symbols) — Apple's built-in library of 5,000+ icons accessible with Image(systemName:) — and Asset Images — your own photos or graphics stored in the Xcode Assets catalog, loaded with Image("name").

SF Symbols are especially useful for beginners because you get professional icons instantly without needing design files.

Swift — Image Examples

Asset image from project Image("profile_photo") .resizable() // ← must add this before frame for images .scaledToFit() .frame(width: 80, height: 80) // 4. Circular profile image Image("profile_photo") .resizable() .scaledToFill() .frame(width: 80, height: 80) .clipShape(Circle()) // ← clips to a perfect circle // 5. Icon with padding and background (icon badge) Image(systemName: "bell.fill") .font(.title2) .foregroundColor(.white) .padding(12) .background(Color.orange) .cornerRadius(14) // 6. Tinted SF Symbol (iOS 15+) Image(systemName: "checkmark.circle.fill") .symbolRenderingMode(.multicolor) .font(.system(size: 40))

🔘 Button

What is Button?

The Button view creates a tappable element that triggers an action when the user taps it. Every Button has two parts: a label (what it looks like) and an action (what it does when tapped). The action is a closure — a block of code that runs on tap.

Swift — Button Examples
// 1. Simplest button — string label shorthand Button("Tap Me") { // code to run when tapped print("Button was tapped!") } // 2. Full syntax — label is any View Button(action: { print("Tapped!") }) { Text("Submit") } // 3. Styled button — primary action style Button("Sign In") { print("Signing in...") } .font(.headline) .foregroundColor(.white) .frame(maxWidth: .infinity) .padding() .background(Color.blue) .cornerRadius(14) // 4. Outline / ghost button style Button("Cancel") { } .font(.headline) .foregroundColor(.blue) .frame(maxWidth: .infinity) .padding() .background(Color.clear) .overlay( RoundedRectangle(cornerRadius: 14) .stroke(Color.blue, lineWidth: 1.5) ) // 5. Icon button Button(action: { }) { Image(systemName: "heart.fill") .font(.title) .foregroundColor(.red) } // 6. Button with icon + text (HStack label) Button(action: { }) { HStack { Image(systemName: "arrow.right.circle.fill") Text("Continue") } .font(.headline) .foregroundColor(.white) .padding() .background(Color.orange) .cornerRadius(14) }

📝 TextField

What is TextField?

The TextField view provides a user-editable text input area. It requires a binding — a two-way connection to a @State variable — so whatever the user types is stored in your app's state automatically. TextField shows a placeholder (hint text) when the field is empty.

Use SecureField instead of TextField when you need a password field — it hides the text the user types.

Swift — TextField Examples
// TextFields require @State for two-way binding struct LoginForm: View { // @State stores the current text input values @State private var name: String = "" @State private var email: String = "" @State private var password: String = "" var body: some View { VStack(spacing: 16) { // 1. Basic text field TextField("Enter your name", text: $name) // 2. Styled text field TextField("Email address", text: $email) .textFieldStyle(.roundedBorder) .keyboardType(.emailAddress) // 3. Password field (hidden text) SecureField("Password", text: $password) .textFieldStyle(.roundedBorder) // 4. Custom styled field TextField("Search...", text: $name) .padding(12) .background(Color(.systemGray6)) .cornerRadius(10) } } }
Swift — Live Binding Demo
struct GreetingView: View { @State private var userName = "" var body: some View { VStack(spacing: 20) { TextField("Type your name...", text: $userName) .padding() .background(Color(.secondarySystemBackground)) .cornerRadius(10) // This text updates as the user types! Text(userName.isEmpty ? "Start typing above..." : "Hello, \(userName)! 👋") .font(.title2) .bold() } } }

⚠️ TextField always needs a @State binding. If you try to create a TextField without a $variable binding, Xcode will show a compile error. The $ prefix means "binding" — a two-way connection between the field and the variable.

Section 05 · Watch Out For

Common Beginner Mistakes

Every developer makes these mistakes — especially when starting out. Learning to recognize them early will save you hours of debugging frustration in your labs.

🔴Applying modifiers in the wrong order
The most common SwiftUI mistake. Remember: modifiers apply from top to bottom, and each wraps the current state of the view. The most important rule: .padding() must come before .background() if you want the background to fill the padded area. And .cornerRadius() must come after .background() to round the filled shape.
🔴Forgetting @State on a TextField binding variable
If you declare var name = "" instead of @State private var name = "", Xcode will show a compile error when you try to use $name in a TextField. The $ binding syntax only works with @State (and other property wrappers). Always use @State private var for any variable tied to a TextField.
🔴Using Image without .resizable() when sizing
If you try to use .frame(width: 80, height: 80) on an Image without first calling .resizable(), the image won't resize — it will stay at its original size and either overflow or show incorrectly. Always add .resizable() as the first modifier on any Image you want to resize, followed by .scaledToFit() or .scaledToFill().
🔴Confusing Text (display) with TextField (input)
Text is read-only — it displays information but users cannot type in it. TextField is editable — users can type into it. A common mistake is trying to use Text for user input or adding unnecessary bindings to Text views. If users need to type: use TextField. If you're showing information: use Text.
🔴Missing the $ prefix in TextField binding
Writing TextField("Name", text: userName) instead of TextField("Name", text: $userName). The dollar sign $ creates a binding (a two-way connection) between the TextField and the state variable. Without it, the TextField cannot write changes back to the variable. Xcode will show "Cannot convert value of type 'String' to expected argument type 'Binding<String>'"
🔴Not using .font() to size SF Symbol images
SF Symbol icons (Image with systemName) are treated like text — their size is controlled by .font(), not .frame(). For example: Image(systemName: "star.fill").font(.largeTitle) makes a large icon. Using .frame(width: 50) alone won't resize an SF Symbol the way you might expect. For non-symbol (asset) images, use .resizable() + .frame().
🔴Button action closure is empty or missing
Buttons require an action closure — even if it's empty for now: Button("Save") { }. The curly braces cannot be omitted completely. Also, beginners sometimes confuse the label and action positions. Always remember: Button("label text") { /* action here */ }. If the button should navigate or do something later, put a print() statement inside as a placeholder so you can verify the button works in the simulator.
🔴Expecting UI to update without state
This is fundamental to SwiftUI: if you change a regular variable, the UI will NOT update. SwiftUI only re-renders views when @State (or other tracked properties like @ObservedObject) change. If your UI isn't updating when you expect it to, the first thing to check is whether the variable causing the update is marked with @State.
🧠 Mistake Recognition Quiz
A student writes this code and complains "my Image is huge and taking over the screen!" What is the most likely fix?
Image("banner_photo") .frame(width: 100, height: 100)
✅ Correct! Asset images are not resizable by default. You must add .resizable() as the first modifier, then add .scaledToFit() or .scaledToFill(), and then .frame() to constrain the size. Without .resizable(), the image ignores the frame entirely.
❌ Not quite. The fix is to add .resizable() before .frame(). Without it, SwiftUI ignores the frame size for images. The corrected code: Image("banner_photo").resizable().scaledToFit().frame(width: 100, height: 100)

A SwiftUI struct conforming to the View protocol. Has a body property that returns the visual content. Everything visible on screen is a View.

Modifier

Click to define
A method called on a View that returns a new transformed version of that View. Applied using dot syntax. Order affects the result.

@State

Click to define
A property wrapper that tells SwiftUI to watch this variable. When it changes, SwiftUI automatically re-renders the affected views. Required for TextField bindings.

Binding ($)

Click to define
A two-way connection between a value and a view. The $ prefix creates a binding from a @State variable. Used with TextField to link the field to a variable.

SF Symbols

Click to define
Apple's library of 5,000+ icons built into iOS. Loaded with Image(systemName: "name"). Sized using .font(). Free to use in all Apple apps.

body property

Click to define
The required computed property in every SwiftUI View. It returns "some View" — the description of what should appear on screen. SwiftUI calls this to render the interface.
🏆 Final Review Quiz
You want to display a system icon of a star that is large, colored gold, has padding around it, and sits on a dark background with rounded corners. What is the CORRECT modifier order?
✅ Perfect! The correct sequence: 1) .font() to size the icon, 2) .foregroundColor() to colour it gold, 3) .padding() to add space around it, 4) .background() to fill the padded area with dark colour, 5) .cornerRadius() to round the filled background. This is the standard pattern you'll use hundreds of times in MAC221.
❌ Not quite. The correct order is: .font() → .foregroundColor() → .padding() → .background() → .cornerRadius(). Remember: size and colour the view first, then add space (padding), then fill (background), then round the filled background (cornerRadius).
