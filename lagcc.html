01 · Concept Overview

What is MVVM?

A design pattern that separates concerns into three distinct layers.

VIEW
SwiftUI Views — renders UI, observes ViewModel state
⟵ binds / observes · triggers actions ⟶
VIEW MODEL
ObservableObject — business logic, state, @Published properties
⟵ requests data · receives results ⟶
MODEL
Pure Swift structs / data types — no UI dependency
↑ fetches from
SERVICE LAYER
DatabaseService — simulates DB / API calls
💡 In SwiftUI, the ViewModel uses @Published + ObservableObject, and Views observe it with @StateObject or @ObservedObject.

03 · Model Layer

Course.swift

📁 Models group — Model
Course.swift// MARK: - Course Model
// Pure Swift struct — no UIKit / SwiftUI imports needed
// Identifiable allows use in ForEach loops directly

import Foundation

struct Course: Identifiable {
    let id   = UUID()
    let name       : String  // "iOS App Development with SwiftUI"
    let code       : String  // "MAC221"
    let time       : String  // "1:00 – 2:15 PM"
    let room       : String  // "E-201"
    let instructor : String  // "Prof. Kassou"
}
✅ Key MVVM rule: Models are pure data. They do NOT know about any View or ViewModel. They only describe what the data is, never how it is displayed.

Why Identifiable?

SwiftUI's ForEach requires each element to be uniquely identified. Adding Identifiable with a let id = UUID() property satisfies that requirement automatically — no need to write ForEach(items, id: \.self).

04 · Model Layer

DaySchedule.swift

📁 Models group — Model
DaySchedule.swift// MARK: - DaySchedule Model
// Groups a weekday name with its list of courses

import Foundation

struct DaySchedule: Identifiable {
    let id      = UUID()
    let dayName : String    // e.g. "Monday"
    let courses : [Course]  // zero or more courses that day
}

Composition over inheritance

DaySchedule contains an array of Course structs. This is classic Swift value-type composition — simple, predictable, and copy-safe.

Empty Friday is intentional

The courses: [] case on Friday is handled gracefully in ScheduleView by showing a "No classes" message — the Model itself stays clean.

05 · Service Layer

DatabaseService.swift

📁 Services group — Service
DatabaseService.swift// MARK: - DatabaseService
// Singleton — replace internals with real API/DB when ready

import Foundation

final class DatabaseService {

    static let shared = DatabaseService()   // Singleton instance
    private init() {}                            // Prevents external init

    func fetchSchedule(
        for studentId: String,
        completion: @escaping (Result<[DaySchedule], Error>) -> Void
    ) {
        // Simulate async network / DB latency
        DispatchQueue.global().asyncAfter(deadline: .now() + 0.5) {
            let data: [DaySchedule] = [
                DaySchedule(dayName: "Monday", courses: [
                    Course(name: "College Algebra", code: "MAT115",
                           time: "9:00–10:15 AM", room: "E-215",
                           instructor: "Prof. Kassou"),
                    // … more courses
                ]),
                // … more days
            ]
            completion(.success(data))
        }
    }
}
📌 Singleton pattern: static let shared + private init() ensures a single shared instance. Later, swap the mock data for a real URLSession or CoreData call without touching the ViewModel or Views.
06 · ViewModel Layer

ScheduleViewModel.swift

📁 ViewModels group — ViewModel
ScheduleViewModel.swift// MARK: - ScheduleViewModel

import Foundation
import Combine

final class ScheduleViewModel: ObservableObject {

    // @Published → SwiftUI Views automatically re-render when these change
    @Published var weeklySchedule : [DaySchedule] = []
    @Published var isLoading      : Bool         = false
    @Published var errorMessage   : String?

    func load(for studentId: String) {
        isLoading     = true
        errorMessage  = nil

        DatabaseService.shared.fetchSchedule(for: studentId) { [weak self] result in
            DispatchQueue.main.async {                  // Always update UI on main thread
                guard let self = self else { return }
                self.isLoading = false
                switch result {
                case .success(let data) : self.weeklySchedule = data
                case .failure(let err)  : self.errorMessage = err.localizedDescription
                }
            }
        }
    }
}
🔑 The ViewModel is the only layer that calls the Service. Views never touch DatabaseService directly. The ViewModel translates raw Model data into something the View can bind to.
07 · SwiftUI Binding Mechanics

@Published & ObservableObject

How SwiftUI knows when to redraw the View.

ObservableObject

A protocol that enables a class to broadcast change notifications to SwiftUI. When any @Published property changes, all observing views update.

@Published

A property wrapper that wraps a value and automatically fires an objectWillChange event before the value changes, triggering UI updates.

@StateObject

Used in the owner View (ScheduleView). SwiftUI creates and keeps the ViewModel alive for the View's lifetime. Use @ObservedObject when passing it down.

// In ScheduleView — the View OWNS the ViewModel
struct ScheduleView: View {
    let studentId: String

    @StateObject private var viewModel = ScheduleViewModel()
    //  ↑ SwiftUI manages ScheduleViewModel's lifetime tied to this View

    var body: some View {
        // The view reads viewModel.isLoading, viewModel.weeklySchedule, etc.
        // When those @Published props change → body is called again automatically
    }
}
08 · View Layer

ScheduleView.swift

📁 Views group — View
ScheduleView.swift// MARK: - ScheduleView

struct ScheduleView: View {
    let studentId: String
    @StateObject private var viewModel = ScheduleViewModel()

    var body: some View {
        Group {
            if viewModel.isLoading {
                ProgressView("Loading schedule…")          // State 1: Loading

            } else if let error = viewModel.errorMessage {
                VStack {
                    Text(error).foregroundColor(.red)
                    Button("Retry") { viewModel.load(for: studentId) }  // State 2: Error
                }
            } else if viewModel.weeklySchedule.isEmpty {
                Text("No schedule found.")                  // State 3: Empty

            } else {
                List {                                          // State 4: Data
                    ForEach(viewModel.weeklySchedule) { day in
                        Section(header: Text(day.dayName)) {
                            ForEach(day.courses) { course in
                                CourseRow(course: course)
                            }
                        }
                    }
                }
            }
        }
        .navigationTitle("Weekly Schedule")
        .onAppear { viewModel.load(for: studentId) }
    }
}
09 · View Layer

ContentView.swift

📁 Views group — View · Root screen with NavigationStack
ContentView.swiftstruct ContentView: View {

    // Computed property — presentation logic only, no business logic
    private var todayString: String {
        let fmt = DateFormatter()
        fmt.dateFormat = "EEEE, MMM d, yyyy"
        return fmt.string(from: Date())
    }

    var body: some View {
        NavigationStack {                          // Root navigation container
            VStack(spacing: 20) {

                // ── NavigationLink → SecondScreen (ID Card) ──
                NavigationLink(destination: SecondScreen()) {
                    HStack {
                        Image(systemName: "person.text.rectangle")
                        Text("ID CARD").bold()
                    }
                    .padding().background(Color.blue)
                    .foregroundColor(.white).cornerRadius(10)
                }

                // ── NavigationLink → ScheduleView (date card tapped) ──
                NavigationLink {
                    ScheduleView(studentId: "123456789")
                } label: {
                    Text(todayString).padding()
                        .background(Color.white).cornerRadius(10)
                }
            }
            .navigationBarHidden(true)
        }
    }
}
10 · View Layer — Reusable Components

FlippableCard.swift

📁 Views group — Generic view with @Binding
FlippableCard.swift// MARK: - FlippableCard (Generic View)
// Front and Back can be ANY View type — reusable component

struct FlippableCard<Front: View, Back: View>: View {
    @Binding var isFlipped: Bool   // Owned by parent (SecondScreen)
    var front: Front
    var back:  Back

    var body: some View {
        ZStack {
            front
                .opacity(isFlipped ? 0.0 : 1.0)
                .rotation3DEffect(.degrees(isFlipped ? 180 : 0),
                                   axis: (x: 0, y: 1, z: 0))
            back
                .opacity(isFlipped ? 1.0 : 0.0)
                .rotation3DEffect(.degrees(isFlipped ? 0 : -180),
                                   axis: (x: 0, y: 1, z: 0))
        }
        .animation(.easeInOut(duration: 0.7), value: isFlipped)
        .onTapGesture { isFlipped.toggle() }
    }
}
// IDCardFront.swift and IDCardBack.swift are separate view files
// in the Views group — each displays one face of the card
🔗 @Binding lets FlippableCard read and write isFlipped that is owned by SecondScreen via its @State. This is SwiftUI's two-way data binding at work.

12 · SwiftUI Property Wrappers

Wrappers Used in This App

Each wrapper has a specific ownership and scope rule.

@Published

Used in ScheduleViewModel on weeklySchedule, isLoading, errorMessage. Marks a property to broadcast changes to any observing SwiftUI view.

@StateObject

Used in ScheduleView. Creates and owns a ScheduleViewModel instance. SwiftUI keeps it alive across re-renders of the same view.

@State

Used in SecondScreen / ThirdScreen for flipped: Bool. Local, view-owned state. Not shared outside the view that declares it.

@Binding

Used in FlippableCard for isFlipped. A two-way reference to a parent's @State — reads and writes the parent's value.

📌 Rule of thumb: @State = local · @Binding = borrowed · @StateObject = owns the VM · @ObservedObject = references the VM · @Published = triggers re-render
13 · Xcode Setup

Step-by-Step in Xcode

How to create the Group structure and move the files.

①

Create Groups

Right-click the project folder in the Navigator → New Group. Create four groups: Models, ViewModels, Views, Services.

②

Add New Swift Files

Right-click each group → New File → Swift File. Name them exactly as shown: Course.swift, DaySchedule.swift, ScheduleViewModel.swift, DatabaseService.swift, etc.

③

Move / Refactor Existing Code

Cut each // MARK: section from the original monolithic file and paste into the new dedicated file. Keep the original ContentView.swift in the Views group.

④

Build & Verify

Press ⌘ B to build. Each type is now in its own file — Swift's access control defaults (internal) mean all types in the same module can see each other without imports.
