Slide 01 — Foundations

What is MVVM?

MVVM stands for Model – View – ViewModel. It's a pattern that separates your app into three clear layers so your code stays organized, readable, and easy to change.

📱
View
The screen the user sees. Built with SwiftUI, it only knows how to display data. It never stores or calculates anything.
SwiftUI structbody property
⚙️
ViewModel
The brain. It holds the app's state and logic. When the user taps a button, the View tells the ViewModel, which does the work.
ObservableObject@Published
🗄️
Model
Plain Swift data — just a struct. No UI, no logic. It only describes what a piece of data looks like.
structproperties
🍎 Why does SwiftUI love MVVM? SwiftUI's @Published and @ObservableObject are built exactly for this pattern. When data changes in the ViewModel, SwiftUI automatically refreshes the View — no manual UI update code needed.
Slide 02 — The Model Layer
🟢 Beginner

The Model

The Model is the simplest layer. It's a Swift struct that holds your data. In our Counter app, the Model stores the count value — nothing else. No UI, no logic.

Counter.swift — Model
// ✅ The Model — just a plain Swift struct.
// No SwiftUI. No logic. Only data.

struct Counter {
    var count: Int = 0
}
How the three layers connect
// The Model just holds the value
var counter = Counter()
print(counter.count) // 0

// The ViewModel owns the Model and
// updates it when the user acts
counter.count += 1
print(counter.count) // 1

// The View reads from the ViewModel,
// which reads from the Model
Text("Count: \(vm.counter.count)")

Three rules for Models

struct
Always use a struct. Structs are safe and simple value types — perfect for holding data like a counter.
No SwiftUI
Never import SwiftUI here. The Counter struct has no idea it will appear on a phone screen.
No logic
Just properties. The Model stores count. Incrementing and decrementing happens in the ViewModel — not here.
Think of it this way: Counter is like a scoreboard card. It just shows a number. The ViewModel is the scorekeeper who changes it.
Slide 03 — The ViewModel Layer
🟢 Beginner

The ViewModel

The ViewModel owns the Counter Model and exposes its data to the View. It also provides functions the View calls when the user taps a button. When @Published changes, SwiftUI refreshes the screen.

CounterViewModel.swift
import Foundation

// Step 1 — Adopt ObservableObject
class CounterViewModel: ObservableObject {

    // Step 2 — Own the Model, mark it @Published
    // Any change to counter tells the View to refresh
    @Published var counter = Counter()

    // Step 3 — Write functions for user actions
    // These are the only way to change the data
    func increment() {
        counter.count += 1
    }

    func decrement() {
        counter.count -= 1
    }

    func reset() {
        counter.count = 0
    }
}

How the layers connect

ObservableObject
This protocol lets SwiftUI know this class can be "watched." The View subscribes to it automatically.
@Published var counter
The ViewModel owns the Model. When counter.count changes, @Published fires and SwiftUI re-renders the View.
Functions
User actions are functions. The View calls vm.increment(). The ViewModel updates counter.count. The View never touches the Model directly.
The chain: Counter (Model) stores the number → CounterViewModel owns and changes it → CounterView displays it.
Slide 04 — The View Layer
🟢 Beginner

The View

The View displays data from the ViewModel — which itself comes from the Model. When the user taps a button, the View calls a ViewModel function. The View never touches the Counter struct directly.

CounterView.swift
import SwiftUI

struct CounterView: View {

    // ① Connect to the ViewModel
    @StateObject var vm = CounterViewModel()

    var body: some View {
        VStack(spacing: 20) {

            // ② Read from ViewModel → Model
            Text("Count: \(vm.counter.count)")
                .font(.largeTitle)

            HStack {
                // ③ Call ViewModel functions on tap
                Button("−") { vm.decrement() }
                Button("Reset") { vm.reset() }
                Button("+") { vm.increment() }
            }
        }
    }
}

The full picture

① @StateObject
Creates the ViewModel. The ViewModel already owns a Counter model inside it.
② vm.counter.count
View → ViewModel → Model. The View reads counter.count through the ViewModel. It never holds the data itself.
③ vm.increment()
View → ViewModel → Model. Tapping "+" calls vm.increment(), which updates counter.count in the Model. @Published fires. View re-renders.
Notice the pattern: Counter struct → owned by CounterViewModel → displayed by CounterView. Each layer only talks to the one next to it.

Slide 06 — Property Wrappers

The Magic Words

SwiftUI uses special annotations called property wrappers. These four are the most important for MVVM. You'll use them in every project.

@StateObject
Creates and owns a ViewModel. Use this in the View that first creates it. SwiftUI keeps it alive even if the View redraws.
@StateObject var vm = CounterViewModel()
Use in the parent view
@ObservedObject
Watches a ViewModel created elsewhere. Use this in child views that receive the ViewModel from a parent.
@ObservedObject var vm: CounterViewModel
Use in child views
@Published
Put this on a ViewModel property. When the value changes, SwiftUI automatically refreshes any View that reads it.
@Published var count: Int = 0
Inside the ViewModel class
@State
For simple, local state only one View cares about — like whether a sheet is open. Don't share it across views.
@State private var showSheet = false
Local view state only
Slide 07 — Common Mistakes

Do's & Don'ts

These are the most common MVVM mistakes beginners make. Knowing them now saves hours of debugging later.

✅ Do This

✓
Use @StateObject to create your ViewModel in the root View.
✓
Mark every property the View needs to watch with @Published.
✓
Call ViewModel functions from buttons: Button("Add") { vm.addItem() }
✓
Keep Models as plain struct with only data properties.
✓
Put all calculations and conditions in the ViewModel, not in body.

❌ Don't Do This

✗
Don't use @ObservedObject to create a ViewModel — it can reset unexpectedly, losing all your state.
✗
Don't put logic in the View — writing if count > 5 in body breaks the pattern.
✗
Don't import SwiftUI in the ViewModel — it only needs Foundation.
✗
Don't set data directly from the View — avoid vm.count = 5; call vm.setCount(5) instead.
✗
Don't create the ViewModel inside body — it resets on every render.
QuestionAnswerWhy
Where does logic go?ViewModelKeeps View clean and testable
Where does data shape go?ModelSingle source of truth for data structure
Where does layout go?ViewSwiftUI views are for display only
Who creates the ViewModel?@StateObjectStable lifetime — won't be recreated

01

Name Badge App

The user types their name and a live greeting updates on screen. Simple two-way binding with one ViewModel property.

BadgeViewModel.swift
import Foundation

class BadgeViewModel: ObservableObject {

    @Published var name: String = ""

    // Computed — no @Published needed
    var greeting: String {
        name.isEmpty
            ? "Hello, stranger!"
            : "Hello, \(name)!"
    }
}
BadgeView.swift
import SwiftUI

struct BadgeView: View {
    @StateObject var vm = BadgeViewModel()

    var body: some View {
        VStack(spacing: 16) {
            // $ = two-way binding
            TextField("Enter your name",
                       text: $vm.name)
                .textFieldStyle(.roundedBorder)

            Text(vm.greeting)
                .font(.title)
        }
        .padding()
    }
}

What you'll learn

$vm.name
The $ prefix creates a two-way binding. Typing in the field updates vm.name automatically.
computed var
greeting is calculated from name. It doesn't need @Published because it's derived.
🎯 Your task: Add a "Clear" button that calls vm.clearName(). Write that function in the ViewModel. It should set name = "".
02

Simple To-Do List

Add & delete items in a list. Introduces the Model layer, Identifiable, ForEach, and working with arrays in a ViewModel.

TodoItem.swift — Model
import Foundation

// A plain Model struct — just data
struct TodoItem: Identifiable {

    let  id:     UUID   = UUID()
    var  title:  String
    var  isDone: Bool   = false
}

// Identifiable lets ForEach track each row
// UUID gives each item a unique identity
TodoViewModel.swift — ViewModel
import Foundation

class TodoViewModel: ObservableObject {

    // Array of Model items — published
    @Published var items: [TodoItem] = []

    func addItem(title: String) {
        items.append(TodoItem(title: title))
    }

    func deleteItem(at offsets: IndexSet) {
        items.remove(atOffsets: offsets)
    }

    func toggleDone(item: TodoItem) {
        if let i = items.firstIndex(
                        where: { $0.id == item.id }) {
            items[i].isDone.toggle()
        }
    }

    // Computed — counts remaining items
    var remainingCount: Int {
        items.filter { !$0.isDone }.count
    }
}
TodoView.swift — View
import SwiftUI

struct TodoView: View {
    @StateObject var vm       = TodoViewModel()
    @State private var newTitle = ""

    var body: some View {
        NavigationStack {
            List {
                // Input row
                HStack {
                    TextField("New item...", text: $newTitle)
                    Button("Add") {
                        vm.addItem(title: newTitle)
                        newTitle = ""
                    }
                }
                // Item rows
                ForEach(vm.items) { item in
                    HStack {
                        Text(item.title)
                            .strikethrough(item.isDone)
                        Spacer()
                        Button("Done") {
                            vm.toggleDone(item: item)
                        }
                    }
                }
                .onDelete(perform: vm.deleteItem)
            }
            .navigationTitle("\(vm.remainingCount) left")
        }
    }
}
🎯 Your task: Add a "Clear All" button that removes every item from the list. Write func clearAll() in the ViewModel.
03

Profile Form with Input Validation

Multiple @Published properties, computed validation logic, a disabled Save button, and an alert on success. Real-world ViewModel design.

ProfileViewModel.swift
import Foundation

class ProfileViewModel: ObservableObject {

    // Multiple published fields
    @Published var firstName: String = ""
    @Published var lastName:  String = ""
    @Published var email:     String = ""
    @Published var isSaved:   Bool   = false

    // Computed display name
    var fullName: String {
        "\(firstName) \(lastName)"
            .trimmingCharacters(in: .whitespaces)
    }

    // Validation: email must have @ and .
    var isEmailValid: Bool {
        email.contains("@") &&
        email.contains(".")
    }

    // All fields required to save
    var canSave: Bool {
        !firstName.isEmpty &&
        !lastName.isEmpty  &&
        isEmailValid
    }

    func saveProfile() {
        guard canSave else { return }
        isSaved = true   // triggers the alert
    }

    func clearAll() {
        firstName = ""
        lastName  = ""
        email     = ""
        isSaved   = false
    }
}
ProfileView.swift
import SwiftUI

struct ProfileView: View {
    @StateObject var vm = ProfileViewModel()

    var body: some View {
        Form {
            Section("Your Info") {
                TextField("First Name",
                          text: $vm.firstName)
                TextField("Last Name",
                          text: $vm.lastName)
                TextField("Email",
                          text: $vm.email)
            }
            Section("Preview") {
                Text(vm.fullName)
                Text(vm.email)
                    .foregroundStyle(
                        vm.isEmailValid
                            ? .green : .red
                    )
            }
            Button("Save Profile") {
                vm.saveProfile()
            }
            .disabled(!vm.canSave)
        }
        .alert("Profile Saved!",
               isPresented: $vm.isSaved) {
            Button("OK") { vm.clearAll() }
        }
    }
}
🎯 Your task: Add an age text field. Validate that age is a number between 18 and 100. Add var isAgeValid: Bool to the ViewModel and include it in canSave.
